## 一、项目概述

### 1. 项目背景
随着校园生活的丰富，大学生群体中每天都会产生大量的“碎片化需求”（如跨校区代拿快递、软件调试协助、闲置物品流转等）。目前这些需求多依赖微信群或表白墙等渠道发布，存在信息流失快、匹配效率低、且缺乏正向激励机制等痛点。传统的无偿互助难以长久维持，导致许多校园闲置时间和技能未被充分盘活。

### 2. 项目目标
本项目旨在设计并实现一个逻辑闭环、轻量级的任务调度系统。系统引入RPG游戏中的“冒险者公会”模式，将枯燥的互助行为转化为有趣的“发布悬赏”与“接取任务”。通过虚拟积分流转与信誉等级机制，构建一个高效、透明且自带激励属性的校园互助生态，同时锻炼团队对经典信息系统增删改查（CRUD）与状态机流转的工程实现能力。

### 3. 开发环境与条件
- **开发语言：** C++ (采用标准面向对象编程)
- **图形界面：** Qt 框架 / 控制台交互界面 (视具体情况定)
- **数据库系统：** MySQL / SQLite 关系型数据库
- **版本控制与协作：** Git / Gitee 或 GitHub 团队仓库
- **运行环境：** Windows 10/11 操作系统

## 二、需求分析

### 1. 目标用户与使用场景
- **目标用户：** 在校大学生。
- **使用场景：** - **作为委托人：** 同学A因上课无法取急件，登录系统发布一个“代拿快递”的任务，并悬赏 20 积分。
  - **作为冒险者：** 同学B刚好顺路，在“任务看板”接取该任务，完成后经同学A确认，同学B获得 20 积分和对应经验值。

### 2. 核心功能清单
1. **身份与基座管理：** 用户加密注册与登录，个人面板展示公会等级与积分余额。
2. **任务看板模块：** 提供结构化的任务发布表单（标题、描述、赏金）；实现大厅列表的检索与分页展示。
3. **任务流转控制：** 管理任务状态机（待接取 -> 进行中 -> 已完成），并防止同一任务被多人重复接取的业务校验。
4. **积分结算模块：** 任务合规完成后，自动执行委托人扣除积分、接单者增加积分的基础数值流转。

### 3. 非功能需求
- **性能要求：** 系统基础查询响应时间应低于 1 秒，确保列表刷新的流畅性。
- **安全要求：** 用户密码不可明文存储（使用基础哈希算法）；利用基础 SQL 参数化查询防范简单的 SQL 注入。
- **兼容性要求：** 系统能够在标准的校园机房环境及个人 PC 上无需复杂配置即可编译运行。

## 三、技术路线

### 1. 总体架构
系统采用经典的**三层架构（3-Tier Architecture）**模式：
- **表示层 (UI)：** 负责用户交互界面的数据展示与表单收集。
- **业务逻辑层 (BLL)：** 处理核心鉴权、任务状态扭转条件判断及积分计算。
- **数据访问层 (DAL)：** 封装底层 SQL 语句，管理数据库连接，保障数据读写的一致性。

### 2. 主要技术
- **后端支撑：** 摒弃过度封装的复杂框架，使用标准的 C++ 语言与平铺直叙的控制流，确保代码的高可读性和规范性。
- **数据存储：** 遵循数据库第三范式（3NF）设计，将业务流转落实为最基础严谨的 SQL `SELECT`、`INSERT`、`UPDATE` 操作。

### 3. 数据设计初步考虑
- **主要实体或对象：** 用户实体（User）、任务实体（Task）。
- **关键数据表或数据关系：**
  - `User` 表：存储 `uid` (主键)、密码哈希、公会等级、积分余额。
  - `Task` 表：存储 `tid` (主键)、发布者 ID (外键)、接单者 ID (外键)、悬赏额、任务状态位。

## 四、进度安排

### 1. 阶段划分
项目开发周期为 4 周，采用“最小可行性产品（MVP）”敏捷迭代模式：
- **第一阶段：** 需求锁定与底层基建。
- **第二阶段：** 跑通基础账号系统与任务大厅展示（CRUD）。
- **第三阶段：** 核心状态流转与积分结算逻辑攻坚。
- **第四阶段：** 全流程集成联调、Bug修复与文档验收。

### 2. 时间进度表
- **第 1 周（4月17日 - 4月24日）：** 完成 ER 图设计、数据库物理建表及前后端接口规范约定。
- **第 2 周（4月25日 - 5月1日）：** 跑通登录流程，实现“发布悬赏”数据写入与“看板列表”读取。
- **第 3 周（5月2日 - 5月8日）：** 完成任务接单校验与状态切换，跑通经验值/积分加减法结算逻辑。
- **第 4 周（5月8日 - 5月13日）：** 系统封板，全链路跑通测试；撰写答辩 PPT 与结题报告。

### 3. 风险与应对
- **风险：** 前后端数据联调失败或接口不匹配，导致项目进度延误。
- **应对策略：** 第一周强制产出数据接口字典。开发初期前端可先使用硬编码的 Mock 数据（假数据）构建界面，待后端 API 稳定后再进行无缝替换，确保双线并行。
